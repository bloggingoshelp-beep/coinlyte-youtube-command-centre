# CoinLyte Command Centre Handover

This is the main handover document for future Codex, Claude, Gemini, or human maintainers. Read this before changing the app. The goal is not only to understand what files exist, but also why the system is shaped this way and what must stay reliable.

## Product Intent

CoinLyte Command Centre is a private operating system for the CoinLyte YouTube business. It is not a public website and not a generic analytics dashboard.

The app exists to help Kirtish and selected team members answer these operational questions every day:

- What should be researched, recorded, edited, or published next?
- Which video ideas are backed by audience comments, competitor gaps, market news, or analytics?
- Which sources and research briefs support each planner card?
- Who owns the next step?
- Which sponsor or brand conversations are active?
- Is the channel healthy, and what needs fixing?
- What changed after a live-data refresh?

If a future change does not help these questions, be careful before adding it.

## Current Tech Stack

- Frontend: static HTML, CSS, and vanilla JavaScript.
- Hosting: Vercel.
- Server-side functions: Vercel Serverless Functions under `api/`.
- Authentication: signed HTTP-only session cookie, generated by `api/login.js`.
- Shared operational database: Supabase REST API, called only from Vercel server routes.
- Live-data refresh engine: GitHub Actions plus `.github/scripts/refresh.py`.
- YouTube data: YouTube Data API, YouTube Analytics API, and public YouTube RSS feeds.
- Market/news data: RSS feeds and Google News RSS searches.
- AI idea generation: Anthropic Claude API inside the GitHub Actions refresh script.
- Email notifications: Resend through `api/notify.js`.
- Local fallback storage: browser `localStorage`.
- Test runner: Node static smoke tests in `tests/static-smoke.mjs`.

This is intentionally simple. The app should remain easy for AI tools to understand and modify.

## Main User Experience

### Login

The app is private. Users log in with:

- User ID.
- Access code.

Owners are checked against `OWNER_USER_IDS` and `OWNER_ACCESS_CODE` from Vercel environment variables. Team users are checked against hashed codes stored in Supabase `team_users`.

The owner code must never be hardcoded in frontend files.

### Command

The Command screen is the daily owner board. It summarizes priority actions, latest upload context, quick owner decisions, channel health, and next moves.

Important behavior:

- Quick action cards can be dismissed.
- Owner action queue cards can be dismissed.
- Latest upload thumbnail must stay aligned in a 16:9 frame.
- Buttons should jump to the relevant area instead of opening disconnected pages.

### Analytics

Analytics displays channel performance signals from `assets/live-data.js`. It is read-only and comes from the refresh workflow.

### Channel Intelligence

Channel Intelligence is the source of market and audience insight. It includes:

- Channel health.
- Market Intelligence.
- News Radar.
- Competitor Intelligence.
- Community Pulse.
- Video Ideas.
- Action Plan.

Important behavior:

- Market Intelligence can create planner-ready ideas.
- News Radar is source-only. It is for scanning important crypto news, opening source links, dismissing noise, and saving links to Content Planner -> Saved Radar. Do not turn News Radar into another video-idea generator unless Kirtish explicitly asks.
- Competitor Intelligence should show competitor uploads and CoinLyte-fit ideas inspired by them.
- Community Pulse should show comment-led video ideas before raw top comments.
- Dismissed intelligence cards must stay hidden until restored or local/cloud state is cleared.
- Source links must open in a new tab.

### Content Planner

Content Planner is the working board. It has:

- Planner Board.
- Saved Radar.
- Calendar.
- Team Hub.

Planner cards can be created manually or from intelligence. When a card comes from intelligence, it must carry:

- Original source name.
- Source URL.
- Locked research brief.
- Category.
- Priority.
- Stage.
- Optional assignee.
- Optional sponsor linkage.
- Optional editor reference sources. After a row-level Save, these should display as locked source rows with a Source link and a single Edit action, so editors can tell the reference was accepted.

Current stages:

- Ideas.
- Research.
- Recording.
- Editing.
- Scheduled.
- Published.

Note: The app intentionally uses `Research`, not `Scripting`, because Kirtish does not write full scripts.

### Saved Radar

Saved Radar stores source-only news links that should survive the next refresh. It is fed from Channel Intelligence -> News Radar.

Important behavior:

- Saving radar does not create a planner card.
- Saved Radar persists in localStorage, backup JSON, and Supabase board sync.
- Saved Radar is useful for weekly live videos, team research handoff, and audience sharing.

### Brand Deals

Brand Deals has:

- Brand Directory.
- Deal Board.

It tracks sponsor or brand opportunities with category, communication channel, value, currency, deal type, status, deadline, notes, and contact details.

Important behavior:

- Brand records and deal board cards are backed up separately.
- Brand cards must remain color-coded and easy to scan.
- Brand data syncs through the same `/api/board` operational state when Supabase is configured.

### Team Access

Team Access controls who can log in and which high-level app areas they can open.

Important behavior:

- Team users need a User ID and a separate access code.
- The code is hashed in Supabase; it cannot be viewed later.
- Use reset/set code if the team member forgets it.
- Team users can be Active or Paused.
- Paused users should lose access after reload, sync, or next server validation.
- Board access decides which top-level app areas are visible.
- Notification channels can include In-app, Email, WhatsApp, Slack, and Telegram. Today only In-app and Email are implemented; the other channels are stored as future preferences.

### Refresh

Refresh is for live-data rebuilds. It is not the same as board sync.

Refresh Live Data does this:

1. Browser calls Vercel `/api/refresh`.
2. Vercel dispatches GitHub Actions workflow.
3. GitHub Actions runs `.github/scripts/refresh.py`.
4. The script fetches YouTube, analytics, comments, competitor RSS, news RSS, and Claude-generated ideas.
5. The workflow commits `assets/live-data.js` and `assets/refresh-status.json`.
6. The workflow optionally calls the Vercel deploy hook.
7. Vercel deploys the updated static data.

Sync Board does this:

1. Browser calls Vercel `/api/board`.
2. Vercel reads/writes Supabase `app_state`.
3. Planner, brands, saved radar, team users, notifications, and dismissals update across devices.

Sync Board does not call YouTube, RSS, Claude, GitHub Actions, or Vercel deploy hooks.

## Data Storage Model

The app has three layers of data.

### Layer 1: Static Seed Data

File: `assets/data.js`

This is fallback/default data that lets the app render before live refresh or cloud sync.

### Layer 2: Refreshed Intelligence Data

Files:

- `assets/live-data.js`
- `assets/refresh-status.json`

These are generated by GitHub Actions. They include channel stats, analytics snapshots, competitor videos, comments, news items, market intelligence, video ideas, and refresh metadata.

Do not store planner work here. A refresh should not wipe planner cards, team users, brand deals, or saved radar.

### Layer 3: Operational Board Data

Primary store when configured: Supabase table `app_state`, row key `operational`.

Fallback store: browser `localStorage`.

This stores the active business operation:

- Planner pipeline cards.
- Team Hub links.
- Brand records.
- Saved Radar links.
- Team members.
- Notifications.
- Dismissed intelligence cards.
- Dismissed command cards.

## localStorage Keys

These keys are used as fallback/offline state and as a local cache. Do not rename them without a migration.

- `cl_pipeline_v4`: planner cards.
- `cl_hub_links_v3`: Team Hub links.
- `cl_brands_v3`: brand directory and deal board data.
- `cl_saved_radar_v1`: saved source-only news radar links.
- `cl_dismissed_ideas_v1`: dismissed market, competitor, community, radar, or video-idea cards.
- `cl_dismissed_command_v1`: dismissed Command dashboard cards.
- `cl_team_members_v1`: local fallback copy of team members.
- `cl_notifications_v1`: in-app notification items.
- `cl_refresh_job_v1`: last queued refresh job status.

Supabase is the source of truth when configured. localStorage is still useful for fast render and fallback.

## Supabase Schema

The app uses two tables.

### `app_state`

Stores JSON state for the shared operational board.

Important row:

- `key = operational`
- `data = { pipeline, hubLinks, brands, savedRadar, teamMembers, notifications, dismissedIdeas, dismissedCommand }`

### `team_users`

Stores secure team login data:

- `id`
- `name`
- `user_id`
- `email`
- `role`
- `access_status`
- `access`
- `channels`
- `notify_stages`
- `access_code_salt`
- `access_code_hash`
- timestamps

The access code is hashed with PBKDF2 in `api/auth.js`. It is not recoverable. Reset it if needed.

### Security Model

- RLS is enabled.
- Frontend never talks directly to Supabase.
- Vercel API routes use `SUPABASE_SERVICE_ROLE_KEY`.
- Do not expose `SUPABASE_SERVICE_ROLE_KEY` to frontend code.
- Keep Supabase anon/client-side usage disabled unless a full security redesign is done.

## Vercel Environment Variables

Set these in Vercel Project -> Settings -> Environment Variables.

### Required for Login

- `OWNER_ACCESS_CODE`: owner login access code.
- `OWNER_USER_IDS`: optional comma-separated owner IDs. Example shape: `mrvyas,owner,kirtish`.
- `AUTH_COOKIE_SECRET`: long random string used to sign session cookies.

### Required for Full Refresh

- `GH_PAT`: GitHub token used by Vercel API to dispatch workflow and check runs.
- `GH_OWNER`: GitHub owner. Defaults in code to the current owner if omitted.
- `GH_REPO`: GitHub repo. Defaults in code to the current repo if omitted.
- `GH_WORKFLOW`: optional, defaults to `refresh.yml`.
- `GH_REF`: optional, defaults to `main`.

### Required for Shared Board and Team Login

- `SUPABASE_URL`: Supabase project URL.
- `SUPABASE_SERVICE_ROLE_KEY`: Supabase secret/service key.

### Required for Email Notifications

- `RESEND_API_KEY`: Resend API key.
- `NOTIFY_FROM`: verified sender. Example format: `CoinLyte Command <notification@coinlyte.com>`.

Do not put angle brackets around the whole value. The format is display name plus email in angle brackets.

## GitHub Actions Secrets

Set these in GitHub Repo -> Settings -> Secrets and variables -> Actions.

- `YT_API_KEY`: YouTube Data API key.
- `YT_CLIENT_ID`: Google OAuth client ID.
- `YT_CLIENT_SECRET`: Google OAuth client secret.
- `YT_REFRESH_TOKEN`: Google OAuth refresh token for YouTube Analytics/comment access.
- `ANTHROPIC_API_KEY`: Claude API key for generated video ideas.
- `GH_PAT`: token used by the workflow checkout/commit step.
- `VERCEL_DEPLOY_HOOK`: Vercel deploy hook URL, so refreshed files trigger redeploy.

Never commit secret values.

## Refresh Script Responsibilities

File: `.github/scripts/refresh.py`

Main responsibilities:

1. Read secrets from GitHub Actions environment.
2. Exchange Google refresh token for OAuth access token.
3. Fetch CoinLyte channel RSS uploads.
4. Fetch competitor RSS uploads:
   - Coin Bureau official channel.
   - Cyber Scrilla.
   - Bankless.
5. Fetch recent comments when scopes allow it.
6. Fetch YouTube Analytics reports.
7. Fetch crypto/news RSS feeds.
8. Split news into India, regulation, and market/global groups.
9. Build market intelligence and news radar inputs.
10. Call Claude with channel, audience, comment, competitor, analytics, and news context.
11. Write `assets/live-data.js`.
12. Write `assets/refresh-status.json`.

If YouTube comment fetching fails with insufficient scopes, the rest of refresh can still complete. To fix comments properly, regenerate the Google refresh token with the required YouTube scopes.

## Board Sync Responsibilities

File: `api/board.js`

This API reads/writes the operational board in Supabase.

Owner can sync:

- `pipeline`
- `hubLinks`
- `brands`
- `savedRadar`
- `teamMembers`
- `notifications`
- `dismissedIdeas`
- `dismissedCommand`

Team users can only sync fields related to their allowed app areas. This is a server-side safeguard, not only a UI hide/show.

Important:

- If a new board area is added, update `OWNER_FIELDS`, `ACCESS_FIELDS`, backup/export logic, import logic, localStorage persistence, and docs.
- Sync is merge-based by allowed fields.
- Sync should not call the refresh workflow.

## Notifications

### In-App Notifications

In-app notifications are stored in operational board state and visible through the notification UI.

Created when:

- A planner card assigned to a team member changes stage.
- Assignment/stage logic in `assets/app.js` calls notification helpers.

Users can:

- Open card.
- Mark read/unread.
- Dismiss.

### Email Notifications

Email notification flow:

1. Planner card is assigned to a member.
2. Member has Email channel selected.
3. Member has an email address.
4. Member has stage notifications enabled.
5. Card changes stage.
6. Frontend calls `/api/notify`.
7. Vercel sends email through Resend.

Email is optional. If `RESEND_API_KEY` is missing, the API returns `not_configured` and in-app notification still works.

Troubleshooting email:

- Confirm `RESEND_API_KEY` exists in Vercel production env.
- Confirm `NOTIFY_FROM` uses a verified Resend domain/sender.
- Confirm the team member has Email checked.
- Confirm the team member has an email address.
- Confirm the card is assigned to that member.
- Confirm the card actually changed stage after assignment.
- Check Vercel Function logs for `/api/notify`.
- Check Resend logs.

## Access Control

### Owner

Owner login requires:

- User ID listed in `OWNER_USER_IDS`, or default owner IDs.
- `OWNER_ACCESS_CODE`.

Owner has all app access.

### Team

Team login requires:

- Supabase configured.
- Team user created in Team Access.
- Active access status.
- User ID.
- Hashed access code set/reset from the Team Access modal.

Team users see only the app areas selected in their Board Access.

If access changes while a user is already logged in, the app revalidates through `/api/me` and `/api/board` on reload/sync. A stale browser may need reload if it has not contacted the server yet.

## Backups and Recovery

There are three backup layers.

### 1. Code Backup

GitHub is the source of truth for code. Every stable change should be committed. If a future AI tool breaks the app, restore by reverting the bad commit or redeploying an older Vercel deployment.

### 2. Operational Board Backup

Content Planner backup/export includes:

- Planner pipeline.
- Team Hub links.
- Brand records.
- Saved Radar.
- Team members.
- Notifications.
- Dismissed ideas.
- Dismissed command cards.

Use this before major UI or logic changes.

### 3. Brand Backup

Brand Deals has its own backup/import for brand records. This is useful when only sponsor/deal data needs to be preserved or restored.

### Recovery Workflow

If a code change breaks the UI:

1. Revert the bad Git commit or redeploy the previous Vercel deployment.
2. Do not import backup unless operational data is also wrong.

If operational board data is wrong:

1. Use Content Planner -> Import Backup.
2. Confirm replacement only after checking the backup file.
3. Sync Board after import so Supabase gets the restored state.

If live intelligence data is wrong:

1. Check GitHub Actions refresh logs.
2. Fix secrets or refresh script.
3. Run Refresh Live Data again.

## Deployment Flow

Normal code deploy:

1. Edit code locally.
2. Run checks.
3. Commit.
4. Push to GitHub main.
5. Vercel auto-deploys.

Normal data refresh:

1. Owner opens Refresh.
2. Click Refresh Live Data.
3. Vercel queues GitHub Actions.
4. GitHub Actions commits fresh data.
5. Vercel deploy hook publishes fresh data.

Normal board sync:

1. User changes planner/brand/team state.
2. App saves locally.
3. App persists to Supabase.
4. Other devices/users use Sync Board or reload to pull latest shared state.

## Must-Not-Break Rules

- Do not hardcode access codes, API keys, tokens, Supabase keys, Resend keys, or GitHub tokens.
- Do not expose `SUPABASE_SERVICE_ROLE_KEY` in frontend code.
- Do not remove login protection from `api/static.js`.
- Do not rename localStorage keys without migration.
- Do not let Refresh Live Data overwrite planner, brands, team users, saved radar, or notifications.
- Do not turn Sync Board into a full YouTube/news/Claude refresh.
- Source links must open in a new tab.
- Planner cards created from intelligence must keep source link and research brief.
- News Radar must remain source-only unless explicitly redesigned.
- Team access changes must be checked server-side, not only hidden in the UI.
- Team access codes must remain hashed.
- Keep backup/import flows working before large UI changes.
- Keep the Claude-inspired visual direction: premium cards, color-coded boards, compact scan-friendly sections, and operational clarity.

## Known Operational Notes

- Resend can send from a verified custom domain even if the domain's mailbox is hosted elsewhere.
- Hostinger email and Resend sending can coexist if DNS records are configured correctly.
- If Resend tracking subdomain is configured, keep it separate from normal mailbox records.
- GitHub Actions refresh can succeed even if comment fetching fails; check logs for scope errors.
- YouTube Analytics may not support every dimension for every channel/API version. The refresh script is designed to continue when one report fails.
- Supabase free tier is enough for this app's current operational board scale.

## Troubleshooting

### App Looks Like Plain HTML

Likely CSS is not loading.

Check:

- `/assets/styles.css` returns CSS.
- `api/static.js` includes assets in Vercel.
- Vercel deployment finished.
- Browser cache is cleared or URL has a new `?v=...`.

### Login Fails

Check:

- `OWNER_ACCESS_CODE` exists in Vercel.
- `AUTH_COOKIE_SECRET` exists in Vercel.
- User ID is included in `OWNER_USER_IDS` for owner login.
- Team user is Active.
- Team user has a login code set.
- Supabase env vars exist.
- `supabase/schema.sql` was run.

### Team Access Changes Do Not Reflect

Check:

- User refreshed or synced after access change.
- `/api/me` is reachable.
- `/api/board` is reachable.
- Team user row in Supabase has updated access/status.
- User was not testing from a cached local-only session.

### Board Missing on Another Device

Check:

- `SUPABASE_URL` exists in Vercel.
- `SUPABASE_SERVICE_ROLE_KEY` exists in Vercel.
- `app_state` table exists.
- Sync Board succeeds.
- The first owner session uploaded the local board snapshot.

### Refresh Stuck Queued

Check:

- `GH_PAT` exists in Vercel.
- `GH_PAT` exists in GitHub Actions secrets if workflow needs commit access.
- Workflow name is correct.
- `GH_OWNER`, `GH_REPO`, `GH_WORKFLOW`, `GH_REF` are correct if overridden.
- GitHub Actions run logs.
- `VERCEL_DEPLOY_HOOK` exists in GitHub Actions secrets.

### YouTube Comments Fail

If logs show `insufficient authentication scopes`, regenerate the Google OAuth refresh token with the needed YouTube scopes. The app can still refresh analytics/news/ideas without comments, but Community Pulse will be weaker.

### Email Does Not Send

Check:

- `RESEND_API_KEY`.
- `NOTIFY_FROM`.
- Resend domain verification.
- Team member Email channel.
- Team member email address.
- Assigned card stage movement.
- Vercel `/api/notify` logs.
- Resend event logs.
