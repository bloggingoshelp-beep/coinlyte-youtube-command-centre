Executive Summary

The application is now operating as a real browser-based command centre: users log in with User ID plus access code, team access is controlled from Supabase-backed records, planner and brand boards sync through the server API, and live YouTube/news intelligence is generated through the GitHub refresh workflow.

During this QA pass I found and fixed one important frontend permission issue: restricted tabs were visually hidden, but old hidden DOM content could remain after a team user navigated between areas. I also tightened the static file route so a paused or deleted team user cannot keep using an old signed cookie to fetch the app shell.

No real secret values, access codes, GitHub tokens, Google credentials, Supabase service keys, Anthropic keys, or Resend keys were written into this report or committed frontend files.

Issues Found And Fixed

Restricted hidden DOM content

Finding: a limited team user could only see allowed nav tabs, but old hidden content from restricted sections could remain in the DOM after navigation.

Fix: added restricted-view clearing and unauthorized action scrubbing in the frontend render flow. Restricted nav buttons are now hidden, disabled, and their content is emptied.

Static route used stale cookie check

Finding: static files were protected by signed cookie validation only, not by the latest Supabase team-user status.

Fix: the static route now uses the live session lookup. Paused or removed users are redirected to login and the old session cookie is cleared.

How The Platform Works

Daily app usage

  • Users open the browser app and log in with User ID plus access code.
  • The app asks /api/me who the user is and which high-level boards they can open.
  • The visible tabs are built from that access list.
  • Planner, Brand Deals, Saved Radar, notifications, dismissed items, and team records sync through /api/board and Supabase.
  • Sync Board pulls the newest shared board state without running the full YouTube/news/AI refresh.

Live intelligence refresh

  • Refresh Live Data is an owner workflow, not a normal team sync button.
  • The Vercel API asks GitHub Actions to run the refresh workflow.
  • The workflow reads YouTube, comments, analytics, competitor feeds, news RSS, and Anthropic/Claude generation where configured.
  • The workflow writes assets/live-data.js and assets/refresh-status.json, commits them, and optionally triggers Vercel redeploy through the deploy hook.

  • Planner backup exports the board JSON before risky edits.
  • Brand backup exports brand/deal state separately.

Residual Risks And Honest Notes

  • Email delivery: the server route and required environment variables are in place, but this QA pass did not send a real email to avoid unintended live team communication.
  • Full refresh: the workflow path is in place and previously working, but this QA pass did not trigger a fresh YouTube/news/Claude run because it consumes external quota and may produce production data changes.
  • Per-tab data isolation: current production protects the app behind login and filters board sync fields by user access. If future team roles require strict separation of intelligence datasets, the static live-data asset should be replaced by role-filtered API endpoints.
  • Owner credential test: owner-only backend behavior was verified by code/API protection. Interactive owner login was not repeated in this pass because the current active browser session was a limited team session.
