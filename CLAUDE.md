# CoinLyte Command Centre - Claude Code Instructions

This file is the first thing Claude Code should read when working on this repo.

## Project Role

CoinLyte YouTube Command Centre is Kirtish Vyas' operational dashboard for managing the CoinLyte YouTube business. It is not a static HTML mockup. It is a live owner/team command centre with:

- YouTube refresh data generated by GitHub Actions.
- Shared planner, brand, team, notification, and dismissed-state data stored in Supabase.
- Vercel serverless API routes for auth, shared state, refresh queueing, team users, and notifications.
- A browser-first frontend in `assets/app.js`.

## Product Mental Model

The app is organized into high-level business areas:

- Command: daily owner decision board and quick actions.
- Analytics: read-only channel analytics from refreshed data.
- Channel Intelligence: research and idea intelligence.
- Content Planner: shared video pipeline, calendar, saved radar, team assignment.
- Brand Deals: sponsorship records and deal board.
- Team Access: access permissions, login users, notification preferences.
- Refresh: queue and monitor GitHub refresh workflow.

Channel Intelligence has this structure:

- Market Intel: India Policy, US Regulation, Global Market, and embedded Source Radar.
- Coin Stats: separate Top 30 Coin Momentum tab using CoinGecko top-30 market-cap coins plus fresh 7-day Google News sources.
- Competitor Intel: competitor videos and CoinLyte-fit angles.
- Community Pulse: comment-led demand and suggested ideas, with scam/spam filtering.
- Video Ideas: final ranked shortlist across all intelligence sources.
- Action Plan: operating rules and next moves.

Video Ideas must remain the final combined decision layer. It should consider Market Intel, Coin Stats / Coin Momentum, Competitor Intel, Community Pulse, analytics, saved radar, dismissed ideas, existing planner cards, and recent uploads. Do not turn Video Ideas into a duplicate of any single tab.

## Data Flow

Refresh flow:

1. User clicks Refresh Live Data in the app.
2. `api/refresh.js` queues a GitHub Actions workflow using `GH_PAT`.
3. `.github/scripts/refresh.py` fetches YouTube, competitor, market, comment, analytics, news, and CoinGecko data.
4. The script uses board memory from Supabase when available so already-planned/dismissed topics are not regenerated.
5. The script writes `assets/live-data.js` and `assets/refresh-status.json`.
6. GitHub commits the refreshed files.
7. Vercel deploys the new build through the deploy hook.
8. The app also syncs shared board state from Supabase independently from full live-data refresh.

Shared board flow:

- `api/state.js` reads and writes Supabase `app_state`.
- `team_users` stores hashed team access codes and board permissions.
- Browser local storage is fallback/offline cache only.
- The true shared board state should be Supabase when configured.

## Hard Rules

Do not break these:

- Source links must open in a new tab.
- Planner cards created from intelligence must carry source links and research briefs.
- Saved Radar must survive refresh and can add directly to Planner.
- Dismissed ideas and already-planned topics must not be regenerated as fresh ideas.
- Coin Stats must remain separate from Market Intel.
- Coin Stats must remain tied to top-market-cap coins and the 7-day source window.
- Video Ideas must include Coin Momentum as an input, but must not duplicate the Coin Stats tab.
- Team access must be enforced server-side, not only hidden in the UI.
- Team access codes must remain hashed.
- Supabase service role key must never be exposed in frontend code.
- GitHub, YouTube, Anthropic, Supabase, Resend, and Vercel secrets stay in GitHub/Vercel environment variables only.
- Do not rename localStorage keys unless adding a migration.
- Do not remove duplicate-topic protection from refresh or frontend actions.
- Do not remove scam/spam filtering for Community Pulse.

## Environment Variables

Typical Vercel/GitHub secrets include:

- `OWNER_ACCESS_CODE`
- `OWNER_USER_IDS`
- `AUTH_COOKIE_SECRET`
- `GH_PAT`
- `VERCEL_DEPLOY_HOOK`
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `RESEND_API_KEY`
- `NOTIFY_FROM`
- YouTube OAuth/API secrets used by the GitHub refresh workflow.
- `ANTHROPIC_API_KEY`

Never commit secret values.

## Testing Requirements

Before finishing any code change, run:

```bash
npm test
node --check assets/app.js
PYTHONPYCACHEPREFIX=/private/tmp/coinlyte-pycache python3 -m py_compile .github/scripts/refresh.py
```

For frontend/behavior changes, also browser-test the affected flow:

- Login as owner and team user when access logic changes.
- Sync board after team/access changes.
- Add/dismiss planner cards from Market Intel, Coin Stats, Community Pulse, Competitor Intel, and Saved Radar if touched.
- Verify notifications are visible from the bell/profile area.
- Verify source links open in a new tab.
- Verify board changes persist after reload and sync.

## Style Guidance

The app intentionally uses a premium, colorful, Claude-style dashboard aesthetic:

- High-level navigation should stay simple.
- Sub-tabs should live inside the relevant app area.
- Cards should be color-coded and scan-friendly.
- Market Intel and Coin Stats should feel like intelligence boards, not plain text lists.
- Planner cards should be operational: stage, owner, due date/time, source, research brief, notes, sponsor link, and reference sources.

## Change Discipline

Make one focused change at a time. Avoid broad refactors unless the user asks for them. Preserve working behavior and shared state compatibility. If a change touches refresh logic, verify duplicate filtering and board-memory usage carefully.
